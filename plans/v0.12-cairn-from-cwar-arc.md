# Plan: v0.12 — Cairn From the cwar /plan-Split Arc

**Status:** Draft
**Date:** 2026-04-26
**Driving observation:** Three-PR design loop in cwar (#5 → #7 → #8) produced a portable design package — two new verbs, a folder-as-state convention, a two-destinations disambiguation, and an external-review-as-reframe-generator pattern — that maps directly onto cairn's hollowest pillars (per [MEM feedback/habitat-bootstrap-three-surfaces]).

## Goal

Bring the durable wins from cwar's `/plan` rework into cairn, **gated on a format migration** that fixes the existing slash-invocation reliability issue. Ship together because the format migration is enabling for the new skills.

## Why

Five concrete pulls, each backed by either evidence-on-cwar or a memory entry:

1. **`/note` and `/review` close cairn's pre-launch surfaces.** Cairn currently ships `/reflect` (post-hoc, same-agent) and no pre-emptive intent-capture or external-review verb. Per [MEM feedback/habitat-bootstrap-three-surfaces], that's Pillar 1 (design seeds) hollow and a sub-pillar of Pillar 2 (external-review-as-reframe) hollow. cwar shipped the pattern; we have a worked template.
2. **Folder-as-state beats `STATUS:` fields.** cwar's pre-redesign 49 plans / ~75% drift is a forcing example. Cairn's `plans/` folder will trend the same way — already has 8 files, no terminal lifecycle. The fix is structural, not behavioral.
3. **Two-destinations disambiguation belongs in cairn's skill template.** cwar PR #7 was filed because an agent misrouted a debug finding into cross-session memory when it should have gone to in-repo notes. Cairn's `/note` will face the exact same routing call on day one. Ship the disambiguation table.
4. **SKILL.md subdir format is prerequisite.** [MEM project/cairn-skills-slash-invocation-unreliable] flags that flat `.claude/skills/<name>.md` doesn't reliably register as a slash command. New skills shipped under the broken format would inherit the same bug — wasted release. Format migration is enabling.
5. **External review caught a shipping blocker on this very arc.** cwar's `execute-plan.md` STATUS-field bug only surfaced because a fresh-perspective agent read it cold ([MEM feedback/external-review-as-reframe-generator]). Cairn currently ships nothing analogous. `/review` closes that gap by name.

## Design

### 1. Format migration — `files/skills/<name>.md` → `files/skills/<name>/SKILL.md`

**Pre-flight: verify the format.** The 2026-04-25 memory entry says *"Verify Claude Code's current skill format before shipping."* Before any code change, confirm by:
- Reading current Claude Code docs / changelog for skills
- Testing both flat and subdir formats in a scratch project
- Confirming the bug from [MEM project/cairn-skills-slash-invocation-unreliable] reproduces in flat-format

If the subdir format is wrong, the rest of v0.12 still ships; only this section changes.

**Migration:**
- `files/skills/<name>.md` → `files/skills/<name>/SKILL.md` for all 11 existing skills (advocate, audit, bridge, feedback, plan, prune, reflect, reframe, resume, tour, plus the new ones below).
- Update [adopt.md](adopt.md) install instructions to write to the new paths.
- Update [files/skills/README.md](files/skills/README.md) to document the new layout.
- Local cairn dev: also migrate `.claude/skills/resume.md` → `.claude/skills/resume/SKILL.md`.

### 2. New skill: `/note`

Port from `cwar/.agents/workflows/note.md` with cwar-specific bindings replaced.

**File:** `files/skills/note/SKILL.md`

**Description:** File a quick thought, debug finding, or feature sketch into the project repo. No template, no research phase. Triggers: "note that", "remind me to", "track this", "look at later", bare `/note` after an agent suggestion.

**Distinct from `/reflect`:** `/note` is pre-emptive intent capture (one paragraph, no research, lives in repo). `/reflect` is post-hoc session distillate (memory writes, often user-space). Different gap classes. The skill descriptions must be sharp on the difference.

**Filing destination:** `docs/notes/` in the adopting project's repo, by convention. Configurable per-project but with a sensible default. Filename pattern `NOTE_<TOPIC>_<YYYY-MM-DD>.md` (date in filename so age is visible at `ls`).

**Two-destinations disambiguation:** Ship the table verbatim from cwar's note.md:27-34 (pre-extraction). Header: *"`/note` files to the project repo. Cross-session agent memory is a different verb — file there if the thought shapes future agent reasoning regardless of context."* Adopting agents inherit the rule.

**Promotion path:** `/spec --from <note>` — but cairn has no `/spec` yet. Option A: ship `/spec` too (heavier this release). Option B: promotion path is a no-op in v0.12; document as v0.13 work. **Recommend B** — `/spec` carries 162-line template ceremony and is overkill for cairn's current scope (most cairn-adopters won't run multi-phase Flash handoffs). Revisit if demand surfaces.

### 3. New skill: `/review`

Codify the external-agent-review pattern that produced PR #8.

**File:** `files/skills/review/SKILL.md`

**Description:** Invite an external agent to review a change set with fresh perspective. Distinct from `/reflect` (same-agent post-hoc). Triggers when the work-author wants a check before "ship": "review this PR", "fresh eyes on", "external review of", "did I miss anything in".

**Mechanism:** The skill instructs the agent to:
1. Read the diff (PR or branch range)
2. **Read adjacent unchanged files** that the diff *expects* to be consistent with — this is the blind-spot class per [MEM feedback/external-review-as-reframe-generator]
3. Check for inconsistency-class bugs (renamed concepts not propagated, removed fields still referenced, etc.)
4. Produce a structured review with file:line refs, severity-ordered, with the blocker-vs-quality-of-life split

**Worked example to cite in the skill:** PRs #5/#7/#8 in cwar. The execute-plan.md blocker is the canonical instance.

**What it doesn't do:** Replace `/reflect`. Different lifecycle. `/review` is pre-ship; `/reflect` is post-session. Both can fire on the same change set.

### 4. Folder-as-state for cairn's own `plans/`

Adopt cwar's `specs/` → `archive/` discipline for cairn's plan artifacts.

- `plans/` becomes the active folder
- `plans/archive/` for shipped plan artifacts
- When a release ships, `git mv plans/v0.X-foo.md plans/archive/v0.X-foo.md` in the closing PR
- **No `Status:` line in plan headers anymore.** Folder location is the status. Existing plans get their `Status:` lines stripped on migration.

This is cairn-repo internal discipline; doesn't change what cairn ships to adopters.

**Migration:** All current `plans/*.md` are shipped (v0.3 through v0.10 represent shipped work; v0.11.x didn't have plan files). Move all 8 to `plans/archive/` as part of v0.12. Going forward, only active drafts live in `plans/`.

### 5. CLAUDE.md / LAWS.md callouts

The two-destinations disambiguation is durable enough to surface globally, not just in `/note`:

- Add a one-paragraph callout in `files/CLAUDE.md` (template) pointing at the rule: "Cross-session memory and in-repo notes solve different problems; see `/note` for the disambiguation table."
- No new LAW; the rule is too specific. The skill's description carries the routing guidance.

## Sequencing

Three options, ranked:

1. **Bundle (recommended).** All five sections in v0.12.0. Coherent shipment, the format migration enables the new skills, the cwar-arc story is intact in one release.
2. **Split format / content.** v0.12.0 = format migration only (just the SKILL.md subdir move). v0.12.1 = `/note` + `/review` + folder-as-state. Lower-risk if the format migration has edge cases.
3. **Skip format, ship content under flat.** Add `/note` and `/review` as flat `.md` files knowing they'll inherit the slash-invocation bug; users can still bare-word them. This is the wrong call — it accumulates more files under the broken format.

**Recommend option 1** unless format migration uncovers complications during the pre-flight verification.

## Open questions

1. **Where does `/note` file by default?** cwar uses `docs/planning/notes/`. Cairn's adopters won't all have `docs/planning/`. Options: `docs/notes/`, `notes/`, configurable via a manifest. Probably `docs/notes/` with override capability.
2. **Does `/review` need access to `gh`?** PR-diff reading is much easier with `gh pr diff`. If `gh` isn't installed, the skill falls back to `git diff <base>...HEAD`. Should the skill description note this dependency, or assume it's ambient?
3. **Should `/note` ship its own template file?** cwar has `docs/templates/NOTE_TEMPLATE.md`. Cairn could ship the template under `files/templates/` for adopters to copy. Adds one file but matches cwar exactly. Defer unless needed — the in-skill body description is probably enough.
4. **What about `/spec`?** Argued for deferral above. Revisit if cairn adopters start asking for multi-phase executor handoffs. Today, no signal.
5. **Worktree-aware skill discovery.** The migration assumes Claude Code reads `<project>/.claude/skills/<name>/SKILL.md` for installed skills. Verify that worktree-spawned sessions see the right path.

## What was skipped

- **`/spec` skill.** Per the open-questions analysis above. May come in v0.13 if demand exists.
- **PR-level mechanical gate on planning.** cwar deferred this to a note (see `cwar/docs/planning/notes/NOTE_PLAN_NOTE_SPEC_REVIEW_2026-04-26.md`). Cairn doesn't have an equivalent enforcement surface yet — no `docs-audit.ts` analog. Per [MEM feedback/escalate-rule-to-mechanical-gate], this is the right next stage but needs a host for the check (the cairn audit-skill is reflective, not mechanical). Defer until cairn has the surface.
- **Auto-promotion of long-lived notes.** cwar has `notes:staleness` in docs-audit. Cairn doesn't. Notes-staleness as a separate cairn surface is v0.13+ territory.
- **Replacing existing skills with subdir/SKILL.md** for non-renaming cases (i.e. format-only). Done as part of section 1; called out here for completeness so it's not surprising.

## Validation plan

Before shipping v0.12.0:

1. Format-migration smoke test: install migrated skills into a scratch project, verify `/resume`, `/reflect`, `/note`, `/review` all trigger via slash and bare-word.
2. Re-adoption test: an existing v0.11.x adopter (cwar itself, ideally) runs the upgrade. Verify the format change doesn't break existing skill references in their CLAUDE.md / scripts.
3. `/review` self-test: invoke `/review` on the v0.12 PR itself. The skill should be capable of reviewing its own introduction. If it can't, that's a description-quality bug to fix before merge.

If validation step 3 fails interestingly (the skill misses something a separate manual review catches), file the gap as a `/note` and ship anyway — `/review` is supposed to converge through use, not be perfect at v0.12.0.
