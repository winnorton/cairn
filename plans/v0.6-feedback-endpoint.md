# Plan: v0.6 — Hosted Feedback Endpoint

**Status:** Shipped (Cairn-side); deployment pending.
**Date:** 2026-04-24
**Driving observation:** During cairn's first 48 hours, multiple Cowork adopting agents hit `gh` unavailability and had to fall back to the paste flow (user manually files the issue). Paste fallback works but has real friction — a durable alternative is now justified.

## Goal

Stand up a hosted endpoint that any adopting agent (regardless of shell access, `gh` availability, or environment) can POST structured feedback to. Endpoint files a GitHub issue on cairn's repo and returns the issue URL.

## Why

- **Durability:** paste fallback depends on user paying attention and acting in-session. Losable.
- **Universality:** the endpoint works from any environment the agent can make HTTP calls from.
- **No user friction:** agent POSTs, user hears "filed — here's the link," and continues working.
- **Cross-ecosystem:** a future Gemini/GPT agent adopting cairn (seed tier) can file feedback without needing cairn-specific CLI tools.

## Design

### Three-level degradation in `/feedback` skill

1. **Preferred — POST to hosted endpoint** (`https://cairn.winnorton.com/feedback`).
2. **Fallback — `gh issue create`** if CLI available and endpoint fails/unreachable.
3. **Last resort — paste draft** for user.

### API contract

```
POST /feedback
Content-Type: application/json

{
  "category":     "gap|bug|skill-idea|clarification|docs|design",
  "severity":     "low|medium|high",
  "title":        "<short title>",
  "body":         "<markdown body>",
  "context":      "<optional additional context>",
  "cairn_version": "<e.g. 0.6.0>",
  "environment":   "<e.g. Cowork, Claude Code, claude.ai>"
}

→ 200 { "issue_url": "https://github.com/winnorton/cairn/issues/N" }
→ 429 { "error": "rate_limited", "retry_strategy": "..." }
→ 400 { "error": "invalid_payload", "retry_strategy": "..." }
→ 502 { "error": "github_api_failed", "retry_strategy": "use paste fallback" }
```

### Server responsibilities

1. Validate payload shape.
2. Light PII sanitization (emails, user paths, tokens).
3. Create GitHub issue via bot token (labels: `agent-filed`, `<category>`, `<severity>`).
4. Return issue URL.
5. In-memory rate limit (1–2 req/min per IP — sufficient for v1).

### Stack

- **Runtime:** Node.js 20+.
- **Framework:** Express.
- **GitHub client:** `@octokit/rest`.
- **Hosting:** Google Cloud Run (scales to zero, cheap).
- **Domain:** `cairn.winnorton.com` (custom domain mapping on Cloud Run).

### Repository structure

```
server/feedback-endpoint/
├── README.md           # Deployment guide (GCP + custom domain)
├── package.json        # Node deps
├── index.js            # Express handler
├── sanitize.js         # PII redaction
├── .gcloudignore       # Exclude node_modules, .env
└── .env.example        # GITHUB_TOKEN, GITHUB_REPO
```

Handler lives in the same repo as cairn itself — transparent, forkable, open to pull requests.

### Auth

**v1: no auth + rate limiting.** The endpoint is public. Rate limits (2 req/min per IP) plus the fact that all issues land visibly in a public repo (where spam is obvious) is sufficient defense.

**Future (if abuse):** per-adoption keys generated at install time, or shared rate-limit tokens. Not worth building until there's actual signal.

### Cairn-side wiring

- `manifest.json` gains an `endpoints` section pointing at `https://cairn.winnorton.com/feedback`.
- `files/skills/feedback.md` updated: three-level degradation (endpoint → gh CLI → paste).

## What was skipped

- **Authentication.** No API keys, no OAuth. Rate limits + public repo visibility are the only defense. Revisit only if abuse appears.
- **Database.** No state persistence beyond GitHub issues. The endpoint is stateless.
- **Async processing.** Synchronous: POST → issue created → response. Simpler; Cloud Run handles concurrency.
- **Monitoring.** Cloud Run's built-in logs + GitHub issue arrivals are enough for v1. Add proper observability only if traffic justifies.

## Outcome

Shipped as v0.6.0. Handler code + deployment guide in `server/feedback-endpoint/`. Cairn-side wiring in manifest and skill. User deploys to GCP; at that point feedback endpoint becomes the default path for `/feedback`.

## Decision log

- **Stack: Node.** Chosen over Python for faster cold starts on Cloud Run and wider JS ecosystem familiarity. Octokit's Node client is mature.
- **Custom domain: `cairn.winnorton.com`.** User-owned, branded, memorable.
- **No auth v1.** Friction vs. spam tradeoff; rate-limiting is sufficient protection for a public maintenance project.
- **Endpoint in cairn repo, not separate.** One repo to clone, one README to maintain, forkable as a whole.
