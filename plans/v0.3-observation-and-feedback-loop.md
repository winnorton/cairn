# Plan: v0.3 — Observation & Agent Feedback Loop

**Status:** Proposed
**Date:** 2026-04-24
**Driving critiques:** #1 (decay w/o signal) and #3 (no observation/feedback loop) from the v0.2 review.

## Goal

Give cairn two linked capabilities:
1. **Observation** — agents cite laws/memories inline when applied; an `/audit` skill reports usage frequency; `/prune` gains real signal instead of gut-feel.
2. **Agent-driven feedback** — when an agent using cairn notices a gap, bug, or missing piece, it files feedback directly to cairn without user relay. Feedback is triaged into either immediate execution (tiny fixes) or formal Plan artifacts (design-touching items). The active session user is not required to triage in-session.

## Why

- Without observation, `/prune` is guesswork. A law that fires every session and one that hasn't fired in six months look identical from the files.
- Without agent feedback, cairn improvements depend on users remembering to report friction. Users are not the ones hitting the friction first — agents are. Agents should be the primary reporters.
- The no-user-in-session constraint matters because feedback during active work shouldn't derail the work. File, notify briefly, continue.

## Design

### Part 1 — Observation (citations + `/audit`)

- **Citation convention** baked into `files/LAWS.md` and `files/MEMORY.md` templates: when an agent applies a law or memory, cite inline — `[LAW 3]`, `[MEM user_role]`. Pure text change; no infrastructure.
- **`/audit` skill** scans the current session transcript (and recent transcripts when accessible) for these citations. Reports frequency per law/memory. Surfaces "uncited in N sessions" candidates that feed directly into `/prune`.

### Part 2 — Agent feedback loop

- **`/feedback` skill** invoked when an agent notices a cairn-relevant gap. Produces a structured submission:
  ```
  { category: gap | bug | skill-idea | clarification | docs,
    title, body, severity: low | medium | high,
    suggestedAction, context: { session excerpt, environment } }
  ```
- **Delivery** — prefer `gh issue create` via the `gh` CLI when available (agent uses existing auth, no new infrastructure). Fall back to printing a preformatted issue body the user pastes.
- **Visibility rule** — agent always tells the active session user after filing: "I filed feedback to cairn about X — [link]." Not invisible; just not blocking on them.
- **Scope restriction** — skill description must say "only for gaps in cairn itself, not in the user's own project." Agent judgment filter.

### Part 3 — Triage & Plans-as-artifacts

- **`plans/` directory** (this file lives here — the directory exists as of this plan).
- **Triage path:**
  - **Auto-executable**: tiny, self-contained (typo, unclear sentence, missing bullet). A reviewing agent opens a PR; a maintainer merges.
  - **Plan-required**: anything with design implications. A reviewing agent converts the feedback into a Plan file in `plans/`. Human reviews before implementation.
- **Session user is not in the loop for either path.** Maintainer (repo owner) handles triage async.

2. [R] Write `files/skills/audit.md`.
3. [R] Write `files/skills/feedback.md` (Phase-2 path — uses `gh` CLI or prints-to-paste fallback).
4. [R] Update `manifest.json` and `adopt.md` with new skills.
5. [R] Update README (skills list + a "feedback to cairn" section).

**Ship Phases 1–2 together as v0.3.0.** All [R] (reversible) — pure content and templates, no hosted infrastructure.

### Deferred to later plans

- **v0.4 — Hosted feedback endpoint** [!]: Vercel function accepting POST JSON, creating GitHub issues via bot token. Introduces hosted infrastructure; not reversible as cheaply. Only worth building if v0.3's `gh`-based path shows real volume and real friction.
- **v0.5 — Triage automation** [!]: GitHub Action or cron that classifies new issues, auto-PRs tiny fixes, generates Plan files for larger items. Only worth building if feedback volume justifies automation.

## Assumptions

- Agents running cairn generally have `gh` CLI or web fetch. If not, `/feedback` falls back to printing an issue body for the user to paste — still no user-triage burden, just a copy-paste step.
- Claude Code session transcripts are readable by the agent for `/audit`. Cowork may differ; flagged as open question.
- Citation noise (inline `[LAW 3]` refs) is tolerable. If real users hate it, retire the convention.
- `gh issue create` creates issues against the cairn repo using the agent's/user's existing token. This is safe because they're filing issues against a *public* repo they have access to — not modifying private state.

## Risks & open questions

- **Signal quality** — agents may hallucinate law numbers or forget to cite. `/audit` output is directional, not rigorous. Document this clearly in the skill.
- **PII leakage** — agents could include user info in feedback bodies. `/feedback` skill must include a sanitization rule ("no emails, no keys, no full file paths from the user's machine").
- **User surprise** — "filing without a user" still requires visibility. Skill enforces post-file notification ("I filed X to cairn — [link]").
- **Scope creep in `/feedback`** — agents might file irrelevant stuff. Skill description restricts to cairn-specific gaps; agents should use their project's own issue tracker for project-specific feedback.
- **Cowork parity** — unknown whether Cowork agents can invoke `gh` or similar. May need a Cowork-specific fallback path, or the print-and-paste route handles it.
- **Abuse (future)** — Phase 3's public endpoint invites spam. Explicitly deferred to v0.4 with rate limiting.
