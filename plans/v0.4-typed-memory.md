# Plan: v0.4 — Typed Memory

**Status:** Shipped
**Date:** 2026-04-24
**Driving critique:** Finding 2 from a session-end reflection — citation convention feels awkward for memories, natural for laws.

## Goal

Reorganize memory around the four existing types (`user | feedback | project | reference`)
instead of one flat namespace, because the types already encode different usage character
and the citation convention should align with that rather than pretend memory is uniform.

## Why

- Laws fire discretely; memory diffuses. But memory isn't uniformly diffuse —
  only `user` memory is truly always-on. `feedback` fires like a mini-law; `project`
  shapes specific decisions; `reference` is episodic lookup.
- Flat memory + uniform `[MEM name]` citation forced every type into the worst-fitting
  convention: either performative noise (citing user memory every response) or missing
  signal (skipping citations for memories that really do fire).
- Cairn is <1 day old. Structural change is cheapest right now.

## Design

**Tree:**
```
~/.claude/memory/
├── MEMORY.md
├── user/{README.md, <entries>.md}        # no citation
├── feedback/{README.md, <entries>.md}    # [MEM feedback/<name>]
├── project/{README.md, <entries>.md}     # [MEM project/<name>]
└── reference/{README.md, <entries>.md}   # [MEM reference/<name>]
```

**Per-type conventions** (documented in each subdir's README):

| Type | Character | Citation | Hygiene |
|---|---|---|---|
| `user/` | Always-on background | None | Event-driven (situation changes) |
| `feedback/` | Fires discretely | `[MEM feedback/<name>]` | Citation + semantic |
| `project/` | Decision-shaping | `[MEM project/<name>]` | Time-driven (dates pass) |
| `reference/` | Episodic lookup | `[MEM reference/<name>]` | Integrity (links die) |

**Skill updates:**
- `/audit` — regex updated to `\[MEM (feedback|project|reference)/([a-z0-9_-]+)\]`.
  `user/` excluded from citation audit (it has no citations by design). Report grouped
  by type.
- `/prune` — per-type hygiene logic, one pass per type with different heuristics.

**Forcing function:** if a `feedback` memory fires often enough to cite every time,
promote it to `LAWS.md`. The types create gravitational pulls:
- Always cite + binary compliance → law
- Cite discretely when applied → feedback memory
- Cite when driving a decision → project memory
- Cite at lookup → reference memory
- Never cite (always-on) → user memory

## What was skipped

- Migration logic for existing flat memory files — <1 day since launch, no installed base
  worth protecting. Users who installed v0.3.x can re-adopt; flat files left in place
  will be harmlessly ignored.
- User-type citation pattern — intentionally left out; reinforces the "don't cite" rule.

Shipped as v0.4.0. Memory tree, skills, adopt.md, README all updated in one commit.
