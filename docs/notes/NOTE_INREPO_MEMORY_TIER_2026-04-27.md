# In-repo memory tier (multi-dev scenario) — Note

> **Filed:** 2026-04-27 · **By:** Claude (post-v0.13.0 ship, design discussion with maintainer) · **Status:** open (capturing ahead of the day it happens)

When (not if) a second developer joins cairn or any cairn-adopting project — using the same agent tools — the asymmetry that emerges is that in-repo artifacts (`LAWS.md`, `CLAUDE.md`, `HANDOFF.md`, skills, `adopt.md`) flow via `git pull` but cross-session memory at `~/.claude/projects/<slug>/memory/` does not. The new dev's agent gets the framework but not the project's accumulated lessons (e.g. v0.13.0's `/review`-blind-spot insight, `[MEM feedback/review-catches-explainer-drift]`, would be invisible). The design decision is to add an **in-repo memory tier** at `<repo>/.claude/memory/`, git-tracked and shared across collaborators, alongside the existing user-space tiers — with `/reflect` growing a four-way categorization gate (project-shared / personal-to-this-project / cross-project agent-meta / user-global) that routes new entries to the right home. Existing entries get retrofitted in batches. When real, this is v0.14-shaped (manifest entry, /reflect gate, /resume + /audit + /prune learning the new tier, retrofit, onboarding prompt, citation resolver, audit-mode split, adopt.md migration note). **A precondition the v0.14 work has to address head-on:** many users — even sophisticated ones — don't realize that agent memory at `~/.claude/projects/<slug>/memory/` is local-only, not cloud-synced, not backed up by default, and not visible to anyone else (a separate user-private backup pattern is a partial answer; see fourth bullet). The onboarding prompt has to do double duty: introduce the in-repo tier AND surface enough about user-space storage that adopters can make informed choices about both tiers. Three design corners closed below; four still worth chewing on before spec promotion: forked-project memory divergence (do typed-memory entries merge well or do they pile up like CHANGELOG conflicts?), promotion-to-LAW workflow (probably PR-citing durable-history count as evidence), `/prune` authority on shared memory (one human pruning another's in-repo entries — what's the etiquette?), and `[MEM project/X]` snapshot-supersede hygiene (the `cairn_at_v0.13.0` retiring `cairn_at_v0.12.3` pattern via PR rather than file deletion in user-space).

- **Citation contract — same form, location-disjoint by category.** `[MEM feedback/X]` works as today. Resolver checks in-repo first → user-slug → declared cross-slug pointers (HANDOFF.md `## Related memory paths`). Project-shared and personal-to-this-project don't collide because the categories are disjoint by construction (a fact is *one or the other*, not both). Adopters' existing citations keep working unchanged; only the resolver learns about the new tier.
- **Multi-machine forcing function — citations in commits *are* the bucket.** `[LAW commit-cite]` already requires citations land in durable output, which means git-shared by construction. So `git log --all --grep='\[MEM feedback/'` is the cross-machine count *for free*; `/audit` grows two modes — live-session (per-machine, current behavior) and durable-history (multi-machine, via git). Promotion-to-LAW reads durable-history count, optionally windowed by recency (`--since='3 months'`) so old cold entries don't keep firing the promotion threshold. Don't try to sync per-machine counts; let git aggregate naturally.
- **Onboarding as teaching moment (with local-storage education).** When `adopt cairn` or `/resume` detects an existing `<repo>/.claude/memory/`, agent prompts: *"This project has shared memory — N feedback entries, N project entries, N reference entries the previous collaborators captured. These describe patterns this codebase has run into. Want me to load them as context? (recommended)"* — educates the new dev about the value AND gates consent. Same shape as cairn's existing `adopt` consent gate. **The same prompt should briefly surface where the *user-space* memory tier lives** (`~/.claude/projects/<slug>/memory/`, this user's machine only, not synced, not backed up by default) so the adopter knows the difference between what's shared and what's private. Many adopters won't realize the user-space tier exists until they're told — discovering it on machine loss or a teammate-handoff is the wrong way to find out. Personal preferences stay in user-space; only project-shared facts come from in-repo.

- **Complementary to a user-private backup pattern, not a replacement.** A separate backup system the maintainer has set up (snapshot `~/.claude/`, `~/.gemini/`, project-level `.claude/.gemini/CLAUDE.md` files into a Git repo on cloud storage — at `~/projects/backup-system/backup-ai-configs.sh`, scheduled, with an Ink TUI for monitoring) is the *fail-soft* answer to the same fundamental gap: capture everything user-space, keep it private, recover from machine loss. The in-repo tier is the *fail-loud* answer: only genuinely project-shared entries land in the repo, with strict categorization at `/reflect` time. Different threat models — backup recovers your *personal* state; in-repo tier shares the *project's* state. **Two design dividends from the backup script for v0.14:** (1) the credential exclusion list (defense-in-depth: tar excludes + `.gitignore` covering `credentials.json`, `*.oauth*`, `*.pem`, `*.key`, `id_rsa*`, `token.json`, etc.) should be stolen verbatim as the templated `<repo>/.claude/memory/.gitignore` — repo-visible memory's privacy surface is wider than user-space's, exclusion is non-negotiable; (2) the "meaningful vs. regenerable" split the backup uses (keeps `~/.gemini/antigravity/<memory|skills|knowledge>/`, drops `<browser-profile>/`) is the same surgical instinct `/reflect`'s categorization gate is making — what's worth carrying forward, what isn't. Worth pairing the systems explicitly in v0.14 docs so adopters understand they're solving different (complementary) problems on the same surface.
