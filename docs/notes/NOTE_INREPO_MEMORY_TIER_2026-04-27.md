# In-repo memory tier (multi-dev scenario) — Note

> **Filed:** 2026-04-27 · **By:** Claude (post-v0.13.0 ship, design discussion with maintainer) · **Status:** open (capturing ahead of the day it happens)

When (not if) a second developer joins cairn or any cairn-adopting project — using the same agent tools — the asymmetry that emerges is that in-repo artifacts (`LAWS.md`, `CLAUDE.md`, `HANDOFF.md`, skills, `adopt.md`) flow via `git pull` but cross-session memory at `~/.claude/projects/<slug>/memory/` does not. The new dev's agent gets the framework but not the project's accumulated lessons (e.g. v0.13.0's `/review`-blind-spot insight, `[MEM feedback/review-catches-explainer-drift]`, would be invisible). The design decision is to add an **in-repo memory tier** at `<repo>/.claude/memory/`, git-tracked and shared across collaborators, alongside the existing user-space tiers — with `/reflect` growing a four-way categorization gate (project-shared / personal-to-this-project / cross-project agent-meta / user-global) that routes new entries to the right home. Existing entries get retrofitted in batches. When real, this is v0.14-shaped (manifest entry, /reflect gate, /resume + /audit + /prune learning the new tier, retrofit, onboarding prompt, citation resolver, audit-mode split, adopt.md migration note). Three design corners closed below; four still worth chewing on before spec promotion: forked-project memory divergence (do typed-memory entries merge well or do they pile up like CHANGELOG conflicts?), promotion-to-LAW workflow (probably PR-citing durable-history count as evidence), `/prune` authority on shared memory (one human pruning another's in-repo entries — what's the etiquette?), and `[MEM project/X]` snapshot-supersede hygiene (the `cairn_at_v0.13.0` retiring `cairn_at_v0.12.3` pattern via PR rather than file deletion in user-space).

- **Citation contract — same form, location-disjoint by category.** `[MEM feedback/X]` works as today. Resolver checks in-repo first → user-slug → declared cross-slug pointers (HANDOFF.md `## Related memory paths`). Project-shared and personal-to-this-project don't collide because the categories are disjoint by construction (a fact is *one or the other*, not both). Adopters' existing citations keep working unchanged; only the resolver learns about the new tier.
- **Multi-machine forcing function — citations in commits *are* the bucket.** `[LAW commit-cite]` already requires citations land in durable output, which means git-shared by construction. So `git log --all --grep='\[MEM feedback/'` is the cross-machine count *for free*; `/audit` grows two modes — live-session (per-machine, current behavior) and durable-history (multi-machine, via git). Promotion-to-LAW reads durable-history count, optionally windowed by recency (`--since='3 months'`) so old cold entries don't keep firing the promotion threshold. Don't try to sync per-machine counts; let git aggregate naturally.
- **Onboarding as teaching moment.** When `adopt cairn` or `/resume` detects an existing `<repo>/.claude/memory/`, agent prompts: *"This project has shared memory — N feedback entries, N project entries, N reference entries the previous collaborators captured. These describe patterns this codebase has run into. Want me to load them as context? (recommended)"* — educates the new dev about the value AND gates consent. Same shape as cairn's existing `adopt` consent gate. Personal preferences stay in user-space; only project-shared facts come from in-repo.
